# 🧩 图片压缩参数详解（sharp 版）

本文基于 `sharp` 的实际行为，对以下四个参数做深入拆解：

- `quality`
- `palette`
- `compressionLevel`
- `effort`

重点不是“怎么用”，而是**它们到底在改变什么本质**。

---


# 1️⃣ palette（调色板）——“是否砍颜色”

## 一句话理解
👉 **把几百万种颜色压缩成最多 256 种**

---

## 📌 本质原理：颜色量化（Color Quantization）

开启：

```ts
palette: true
```

会触发：

1. 提取代表颜色（最多 256 种）
2. 建立“调色板”（颜色表）
3. 每个像素改为“索引值”（而不是 RGB）

---

## 🎯 核心变化

| 模式 | 存储方式 | 颜色数量 |
|------|--------|--------|
| 真彩色（默认） | 每像素 = RGB | ~1600万 |
| 调色板模式 | 每像素 = 索引 | ≤ 256 |

---

## ⚠️ 是否有损？

👉 **一定有损（对于照片类图片）**

原因：

- 原图颜色 ≫ 256
- 必须“合并相似颜色”

---

## 🧠 可以这样理解

不是“记录颜色”，而是：

> “挑 256 个代表，然后让所有像素去认亲”

---

## 🧩 适用场景

✅ 非常适合：

- UI 图标
- LOGO
- 扁平插画

❌ 不适合：

- 摄影照片
- 渐变丰富图片（会出现色带）

---


# 2️⃣ quality ——“选颜色的精细程度 / 丢数据程度”

## 一句话理解

👉 在 PNG 中：

- **开启 palette → 控制“颜色挑选质量”**
- **关闭 palette → 基本没影响**

---

## 📌 行为分裂（重点！）

### 🟢 情况 A：palette = true

```ts
png({ palette: true, quality: 80 })
```

👉 此时：

- quality 控制：
  - 颜色选择是否精准
  - 抖动（dithering）程度

📊 效果：

| quality | 表现 |
|--------|------|
| 100 | 选最优的 256 色，最接近原图 |
| 50 | 开始明显失真 |
| 10 | 颜色粗糙 |

---

### 🔵 情况 B：palette = false（默认）

```ts
png({ quality: 0 })
```

👉 此时：

- **不会丢失任何像素数据**
- quality **基本被忽略**

✅ 结果：**完全无损**

---

## 🧠 本质区别

| 参数 | 控制什么 |
|------|--------|
| palette | 是否减少颜色数量（是否有损） |
| quality | 怎么减少（减少得好不好） |

---


# 3️⃣ compressionLevel ——“打包强度（无损）”

## 一句话理解

👉 **不改变内容，只改变“压缩得多狠”**

---

## 📌 原理：DEFLATE（和 ZIP 一样）

PNG 压缩分两步：

```text
像素数据 →（可选 palette）→ 压缩（DEFLATE）
```

compressionLevel 控制第二步。

---

## 📊 参数范围

```ts
compressionLevel: 0 ~ 9
```

| 等级 | 特点 |
|------|------|
| 0 | 不压缩，最快 |
| 1 | 快速压缩 |
| 6 | 默认平衡 |
| 9 | 极限压缩（最慢） |

---

## 🎯 是否影响画质？

👉 **完全不会**

因为：

> 它只是“重新打包”，不是“修改内容”

就像：

- ZIP 压缩等级不同
- 解压后文件一模一样

---

## 🧠 可以这样理解

compressionLevel 做的是：

> “把行李压得更紧，而不是扔掉行李”

---


# 4️⃣ effort ——“算法有多拼命”

## 一句话理解

👉 **控制压缩算法“愿意花多少时间优化结果”**

---

## 📌 和 compressionLevel 的区别

| 参数 | 本质 |
|------|------|
| compressionLevel | 压缩算法强度（固定路径） |
| effort | 算法搜索空间（尝试更多可能） |

---

## 🧪 在不同格式中的作用

### 🟢 PNG（sharp / libvips）

```ts
effort: 1 ~ 10
```

主要影响：

- 滤波器选择（filtering）
- 是否逐行优化

👉 effort 越高：

- 尝试更多滤波策略
- 文件更小
- 更慢

---

### 🔵 WebP

```ts
effort: 0 ~ 6
```

影响：

- 预测模式
- 分块策略
- 编码优化

👉 提升空间：

- 体积 ↓ 1%~5%
- 时间 ↑ 很多

---

## 🎯 是否影响画质？

👉 **不直接影响**

但：

- 在有损模式下
- 高 effort = 更聪明地“丢数据”

---

## 🧠 直觉模型

| 参数 | 类比 |
|------|------|
| quality | 画家画得精不精 |
| palette | 要不要减少颜色 |
| compressionLevel | 行李压缩强度 |
| effort | 打包的人有多较真 |

---


# 🔍 compressionLevel vs effort（深入理解版）

## 🧠 一句话结论

```text
compressionLevel = 在一个固定方案里“榨干空间”
effort            = 在多个方案之间“寻找最优方案”
```

---

## 1️⃣ compressionLevel：固定算法路径内的优化

### 📌 控制对象

👉 **DEFLATE 压缩算法内部行为**

PNG 的无损压缩依赖 DEFLATE，本质包括：

1. LZ77（查找重复数据）
2. Huffman 编码（压缩表示）


### 🧠 提高 compressionLevel 时发生什么？

示例数据：

```text
AAAAABBBBBCCCCCDDDDDEEEEE
```

#### 🔵 低 compressionLevel（如 1）

- 快速扫描
- 只找明显重复
- 使用较简单的编码策略

👉 压缩快，但不极致

#### 🔴 高 compressionLevel（如 9）

- 更大窗口查找重复
- 尝试更多匹配方式
- 构建更优的 Huffman 编码

👉 压缩更小，但更慢

### 🎯 核心特征

- 不改变数据结构
- 不改变压缩策略
- 不尝试新路径
- 只是在既定路径上“压得更狠”

### 🧠 类比

> 同一个行李箱  
> 区别只是：随便塞 vs 拼命压缩空间

---

## 2️⃣ effort：在不同策略之间搜索最优解

### 📌 关键点：PNG 压缩前有“预处理步骤”

👉 **Filter（滤波）**

在压缩前，每一行像素会被转换成“更容易压缩的数据形式”。


### 🧪 示例

原始像素：

```text
[100, 101, 102, 103]
```

经过差分：

```text
[100, +1, +1, +1]
```

👉 更容易压缩（重复性更强）


### 📌 PNG 有 5 种 Filter

- None
- Sub
- Up
- Average
- Paeth

不同 filter 会产生：

👉 **完全不同的数据分布 → 压缩率差异巨大**


### 🧠 effort 在控制什么？

👉 **是否尝试更多 filter 方案组合**


#### 🔵 低 effort

- 使用简单策略
- 可能整张图只用一种 filter

👉 快，但不一定最优

#### 🔴 高 effort

- 每一行尝试多种 filter
- 比较压缩结果
- 选择最优方案

👉 慢，但更小

### 🎯 核心本质

| 参数 | 控制层级 |
|------|--------|
| compressionLevel | 压缩算法内部优化 |
| effort | 压缩前策略搜索 |

---

## 🧭 整体流程（非常关键）

```text
原始像素
   ↓
[选择 filter]        ← effort 控制这里
   ↓
转换后的数据
   ↓
[DEFLATE 压缩]      ← compressionLevel 控制这里
   ↓
输出 PNG
```

---

## 🧠 终极类比（工程视角）

### compressionLevel

你已经决定：

- 用纸箱
- 用某种打包方式

区别只是：

- 普通打包
- 极限压缩（挤到极致）

### effort

你在尝试：

- 用纸箱？
- 用真空袋？
- 换摆放方式？
- 拆分再组合？

👉 然后选最小的一种

---

## 🧾 最终总结

```text
compressionLevel → 局部优化（单一路径）
effort            → 全局优化（多路径搜索）
```

---

## 🧠 额外理解（重要）

在很多情况下：

👉 **effort 对最终体积的影响，可能比 compressionLevel 更大**

因为：

- 选对“表示方式” > 拼命压缩“错误的数据结构”

---


# 🧾 总结（核心认知）

## 🔥 一句话总览

- **palette**：决定“要不要砍颜色”（是否有损）
- **quality**：决定“砍得优不优雅”
- **compressionLevel**：决定“打包压多紧”（无损）
- **effort**：决定“算法有多较真”

---

## 🧠 最重要的结论

✅ PNG 是否有损，只取决于：

```ts
palette: true
```

而不是：

```ts
quality
compressionLevel
effort
```

---

## 🎯 推荐配置

```ts
const options = {
  quality: 80,
  palette: true,
  compressionLevel: 9,
  effort: 10
};
```
